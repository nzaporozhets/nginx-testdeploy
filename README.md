# Deploying nginx on K8s using RBAC+CSRs
## Design considerations

This document describes a baremetal Kubernetes cluster installation and configuration to deploy and serve a static website with nginx using CSRs + K8s RBAC for auth and utilizing cert-manager for the website certificates.
Stack:
- Kubernetes;
- containerd as CRI;
- Calico as a CNI addon;
- MetalLB to provide load balancing functionality since cluster is baremetal;
- nginx ingress controller;
- cert-manager;
- optional: ArgoCD as deployment pipeline.

We are going to configure a limited role for interacting with namespaced resources and bind it to a user derived from CSRs CN to allow use of provisioned certificate for interacting with K8s API/CLI. In the "end user" part we're gonna use the newly created account to install and expose nginx – manually and with help of ArgoCD.

**Assumptions and limitations:**
- For our client certs, `kubernetes.io/kube-apiserver-client` is used as a Signer. This will not allow the certs to be automatically approved (which is outside of the scope of this demo) and would require somebody with appropriate permissions to approve the CSR using K8s API or CLI. External Signers can be configured for the cluster, if desired, to separate the control loops.
- Namespace and ingress are configured by the cluster administrator to be consumed by the development team - `nginxadmins`  and `nginxadmins-argocd` in our example;
- Self-signed cert-manager issuer is used for Nginx endpoints – system would need to be configured to trust our CA. For production installation, integrating cert-manager with an issuer of your choice is recommended. The list of available issuers can be found [here](https://cert-manager.io/docs/configuration/issuers/). letsencrypt was considered, but ruled out, as I currently locked myself out of my Porkbun account and can't configure an ACME challenge :) ;
- Dev team has kubeconfig, we're just adding new ctx. Steps of providing that kubeconfig are omitted – for new teams it can be generated by following the verification steps and cutting out admin context. 
- ArgoCD and ArgoCD CLI installation steps are not covered; only parts relevant to specific configuration. 
- Static nginx site is going to be a default welcome page :) Cluster was not configured with CSI for this exercise.
- ArgoCD permissions may suboptimal and could be stripped down more – the goal was to create a POC of the approach, not map out exact minimal permission set. 

## Installation and configuration steps
### Kubernetes installation

The following steps should be executed on all cluster nodes.

Configure prerequisites (enable forwarding, install dependencies, repo keys, etc.):
```bash
cat <<EOF | sudo tee /etc/sysctl.d/k8s.conf
net.ipv4.ip_forward = 1
EOF
sudo sysctl --system
sudo apt install -y curl gnupg2 software-properties-common apt-transport-https ca-certificates

curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.36/deb/Release.key | sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg
echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.36/deb/ /' | sudo tee /etc/apt/sources.list.d/kubernetes.list
sudo apt-get update
sudo apt-get install -y kubelet kubeadm kubectl
sudo apt-mark hold kubelet kubeadm kubectl
```

Install Docker repo keys and containerd:
```bash
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /usr/share/keyrings/docker-archive-keyring.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/docker-archive-keyring.gpg] \
https://download.docker.com/linux/ubuntu \
$(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt update
sudo apt install -y containerd.io
```

Prepare configuration for containerd:
```bash
sudo mkdir -p /etc/containerd
containerd config default | sudo tee /etc/containerd/config.toml
cat /etc/containerd/config.toml
sudo sed -i 's/SystemdCgroup = false/SystemdCgroup = true/' /etc/containerd/config.toml
sudo systemctl restart containerd
sudo systemctl enable containerd
```

Verify it's running:
```bash
sudo systemctl status containerd
```

Install Kubernetes components:
```bash
sudo apt-get install -y kubelet kubeadm kubectl
sudo apt-mark hold kubelet kubeadm kubectl
sudo systemctl enable --now kubelet
```

Initialize the cluster (**from controller node**):
```bash
sudo kubeadm init --pod-network-cidr=192.168.0.0/24
```

Grab the cluster join command from the output – it will contain required tokens and addressses.
From the worker nodes, join the created cluster (after completing components installation):
```bash
kubeadm join 192.168.122.101:6443 --token <yourtoken> \
        --discovery-token-ca-cert-hash <yourhash>
```

Install CNI addon of your choice - Calico in this example: 
```bash
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.28.0/manifests/calico.yaml
```
### Preparing environment for Nginx installation

>Note: Going forward, `k` is gonna be an alias for the `kubectl` command.

First, let's create a namespace for the dev team to use:
```bash
k create ns nginx-manual
```

Install `cert-manager` – it's gonna be used for issuing the certificates for the hosted website .
```bash
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/download/v1.20.2/cert-manager.yaml
```

create a SelfSigned issuer:
```yaml
#ss_issuer.yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: selfsigned-cluster-issuer
spec:
  selfSigned: {}
```

```
k apply -f ss_issuer.yaml
```

Install an ingress controller of your choice. For this demo we're using an nginx ingress controller:
```
k apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.15.1/deploy/static/provider/baremetal/deploy.yaml
```

Create an Ingress (paying attention to your `ingressClassName`):
```yaml
#ngix-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  annotations:
    cert-manager.io/cluster-issuer: selfsigned-cluster-issuer #This will instruct cert-manager which issuer to use
  name: nginxingress
  namespace: nginx-manual #namespaced to whatever our dev team is supposed to use 
spec:
  ingressClassName: nginx
  rules:
  - host: nginxtest.com
    http:
      paths:
      - pathType: Prefix
        path: /
        backend:
          service:
            name: nginx-service #service name should be confirmed with the end users
            port:
              number: 80
  tls:
  - hosts:
    - nginxtest.com # configures AltName for the generated cert
    secretName: nginx-cert #secret name to store the generated cert
```

```bash
k apply -f nginx-ingress.yaml
```

Once created, you should be able to see the generated certificate in the namespace:
```bash
k get cert -n nginx-manual
NAME         READY   SECRET       AGE
nginx-cert   True    nginx-cert   2d6h
```

#### Preparing RBAC configuration

Create a Role describing desired level of end user access. In our example, devs can create and edit configurations, services, pods and deployments, but are prohibited from deleting those resources or, for example, modifying the Ingresses – as those can be configured to be used by multiple teams based on paths, for exmaple. 
```yaml
#nginxadmin-role.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: nginx-manual
  name: nginxadmin-role
rules:
- apiGroups: [""]
  resources: ["pods", "services", "configmaps"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]

- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]
```

```bash
k apply -f nginxadmin-role.yaml
```

Create a RoleBinding for the new role and desired user name (or group):
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: nginxadm-binding
  namespace: nginx-manual

subjects:
- kind: User
  name: nginxadm
  apiGroup: rbac.authorization.k8s.io

roleRef:
  kind: Role
  name: nginxadmin-role
  apiGroup: rbac.authorization.k8s.io
```

```bash
k apply -f nginxadm-rolebinding.yaml
```

#### Generating a certificate for end user API auth 
Instruct end users to create a key of preferred type (RSA, DSA, ECDSA, etc.) and generate a Certificate Signing Request and provide you with the CSR.
```bash
# To be completed by the end user.
openssl genrsa -out nginxadm.key 2048
openssl req -new -key nginxadm.key -out nginxadm.csr -subj "/CN=nginxadm/O=nginxadmins"
cat nginxadm.csr | pbcopy
#Please provide the results back to ops team (should be in the buffer already)
```

Base64 encode the CSR received from the user and prepare a Kubernetes CSR object:
```bash
cat nginxadm.csr | base64 | tr -d "\n"
```
kcsr.yaml:
```yaml
apiVersion: certificates.k8s.io/v1
kind: CertificateSigningRequest
metadata:
  name: nginxadm
spec:
  request: truncated
  signerName: kubernetes.io/kube-apiserver-client
  expirationSeconds: 864000 #set to desired length, ~10 days in this example
  usages:
  - client auth
```

Create the CSR object in the cluster and approve it:
```bash
k apply -f kcsr.yaml
k get csr
k certificate approve nginxadm
```

Now `kubectl get csr` should show `Approved`:
```bash
 k get csr
NAME            AGE     SIGNERNAME                            REQUESTOR          REQUESTEDDURATION   CONDITION
nginxadm   9m29s   kubernetes.io/kube-apiserver-client   kubernetes-admin   10d                 Approved,Issued
```
Extract the issued certificate and provide it to the end user:
```bash
kubectl get csr nginxadm -o jsonpath='{.status.certificate}'| base64 -d > nginxadm.crt
#optional: verify the resulting cert:
openssl x509 -in nginxadm.crt -text -noout
```
#### Verifying the certificate and RBAC rules
Backing up your kubeconfig before is recommended.
Add new credentials: 
```bash
kubectl config set-credentials nginxadm --client-key=nginxadm.key --client-certificate=nginxadm.crt --embed-certs=true
```

Next, add the context: 
```bash
kubectl config set-context nginxadm --cluster=kubernetes --user=nginxadm
  ```

Verify the created context by some of the following commands:

### End user: configuring kubectl to utilize the certificate
Once you've generated your key, certificate signing request, and obtained a certificate from your Ops team, you can use that certificate for Kubernetes cluster access. 

Let's add new credentials: 
```bash
kubectl config set-credentials nginxadm --client-key=<KEYNAME>.key --client-certificate=<KEYNAME>.crt --embed-certs=true
```

Next, add the context: 
```bash
kubectl config set-context <USER NAME> --cluster=kubernetes --user=<USER NAME>
  ```
  Now you can execute `kubectl` commands against new namespace by either adding the `--context nginxadm` flag to your command like so:
  ```bash
  kubectl apply -f nginx-svc.yaml --context <USER NAME>
```
  
  or configure it as default context:
```bash
kubectl config use-context <USER NAME>  
kubectl config set-context --current --namespace <YOUR NAMESPACE>
```

### End user: deploying namespaced nginx on the configured cluster
Now that you have access to the cluster, you can go ahead and deploy the website.
We're gonna be creating a Deployment:
```yaml
#nginx-deploy.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  namespace: nginx-manual
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
      - name: nginx
        image: nginx:stable
        ports:
        - containerPort: 80
```

```bash
kubectl apply -f nginx-deploy.yaml
```

and a Service to expose that deployment. Service name needs to match the name Ops team configured for Ingress – please coordinate. 
```yaml
#nginx-svc.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
  namespace: nginx-manual

spec:
  selector:
    app: nginx

  ports:
  - port: 80
    targetPort: 80

  type: ClusterIP
```

```bash
kubectl apply -f nginx-svc.yaml
```

### Appendix: Same, but different: using ArgoCD
#### Prerequisites: second ingress
Since we're gonna be using the same port and path, ingress controller will intrenally merge the configurations of our ingresses from manual and argocd namespaces. We're gonna use a different domain name for verification.
```yaml
#nginx-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  annotations:
    cert-manager.io/cluster-issuer: selfsigned-cluster-issuer
  name: nginxingress
  namespace: nginx-argo
spec:
  ingressClassName: nginx
  rules:
  - host: argonginxtest.com
    http:
      paths:
      - pathType: Prefix
        path: /
        backend:
          service:
            name: nginx-service-argo
            port:
              number: 80
  tls:
  - hosts:
    - argonginxtest.com
    secretName: nginx-cert-argo
```
